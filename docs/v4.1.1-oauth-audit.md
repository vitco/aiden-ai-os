## 1.2 — Auth file structure (no secret values)

```
Path:        C:\Users\shiva\AppData\Local\aiden\auth\chatgpt-plus.json
Size:        4,572 bytes
On disk:     { version: 1, iv: <hex>, ciphertext: <hex>, authTag: <hex> }
             (AES-256-GCM, machine-bound key per tokenStore.ts:18-42)
```

**Important correction to the dispatch's premise:** the file is **encrypted**. The `models` field cannot be inspected without code-execution under this user account. After decrypt, the in-memory shape (per `core/v4/auth/tokenStore.ts:64-81`) is:

```ts
interface OAuthTokens {
  provider:       string;     // "chatgpt-plus"
  accessToken:    string;     // bearer JWT
  refreshToken?:  string | null;
  expiresAtMs:    number;     // epoch ms
  account?:       string;
  models?:        string[];   // optional; rarely populated (see below)
  extras?:        Record<string, unknown>;
  savedAt?:       string;
}
```

**Key finding:** `tokens.models` is **type-permitted but never written** by either the OAuth flow or the chatgpt-plus plugin. `core/v4/auth/oauthFlow.ts:474-482` writes `extras` (token-exchange response keys) but not `models`. `core/v4/auth/providerAuth.ts:217` persists `{ provider, accessToken, refreshToken, expiresAtMs, account, extras }` — no `models`. Grep confirms zero callers ever set `tokens.models`. **So the stored token does NOT carry a stale `models` list — that field is empty/undefined for every chatgpt-plus install.**

## 1.5 — Where 4xx errors are surfaced (CRITICAL FINDING)

**File:** `providers/v4/codexResponsesAdapter.ts:349-381`

```ts
const status = response.status;
const raw    = await safeReadBody(response);          // ← body IS read

if (status === 429) { ... throw new ProviderRateLimitError(this.providerName, raw); }
if (status >= 500 && status < 600) {
  throw new ProviderError(
    `Provider ${this.providerName} server error ${status}`,
    this.providerName, status, raw, true,            // ← body attached to .raw
  );
}

// 4xx (auth, content policy, malformed) — fail fast.
throw new ProviderError(
  `Provider ${this.providerName} request failed (${status})`,
  this.providerName, status, raw, false,             // ← body attached to .raw
);
```

`safeReadBody` (defined elsewhere in the file) reads the response body and the result is attached to `ProviderError.raw`.

**The body is NOT discarded.** But the constructor message string (`providers/v4/errors.ts:17-28`) never concatenates `raw` into `.message`:

```ts
export class ProviderError extends Error {
  constructor(
    message: string,
    public readonly providerName: string,
    public readonly statusCode?: number,
    public readonly raw?: unknown,            // ← preserved on the instance
    public readonly retryable: boolean = false,
  ) {
    super(message);                            // ← only the short message
    this.name = 'ProviderError';
  }
}
```

So when the user-facing surface logs `err.message`, they see *"Provider chatgpt-plus request failed (400)"* with no detail — even though `err.raw` contains OpenAI's explanation (e.g. `{"error":{"message":"model 'gpt-5' is not supported when using Codex with a ChatGPT account","type":"invalid_request_error"}}`). **Phase 3 fix is well-targeted: enrich the message string with `parseErrorMessage(raw)` while keeping `.raw` for programmatic consumers.**

## Root cause hypothesis (ranked by likelihood)

### #1 (most likely) — User's persisted active model is `gpt-5` (or another non-`-codex` variant) on a Codex-only-entitled account
- `cli/v4/aidenCLI.ts:617-619` pulls active model from `config.getValue('model.modelId', …)`.
- The catalog still lists `gpt-5` at `modelCatalog.ts:215-227` with its own warning note. Nothing filters the picker by entitlement.
- If the user ever selected `gpt-5` via `/model` (or via a wizard that defaulted to it on a prior version), the choice persisted. Every subsequent `hi` would 400 with "model not supported when using Codex with a ChatGPT account."
- This is consistent with Shiva's report (simple `hi` 400s, README claimed verified at ship — true with `gpt-5.1-codex-max`, regressed when the persisted active-model drifted).
- Note: the OpenAI Feb 13 2026 retirement of `gpt-5`/`gpt-5-mini` is the **upstream API path** retirement. The Codex backend may still recognize `gpt-5` as a slug but reject it for non-entitled accounts — same 400, different code path.

### #2 — Picker offers slugs the user's tier doesn't grant (`gpt-5.5`, `gpt-5.3-codex`, …)
- The picker has no entitlement filter (`cli/v4/commands/modelPicker.ts:80`).
- A user clicking `gpt-5.5` or `gpt-5.3-codex` while on a tier that only entitles `*-codex-max/-mini` would hit the same 400.
- This is the same root-cause class as #1 but triggered by the picker rather than legacy config.

### #3 (unlikely) — Stale `tokens.models` scope
- The dispatch hypothesizes this, but **the field is never written**. The Codex backend doesn't reject based on the token's `models` claim either — its 400 comes from the model slug in the request body matched against the account's server-side entitlement list.
- Effectively rule this out.

### #4 (low) — Payload schema drift in codex Responses API
- `codexResponsesAdapter.ts` already handles the `stream:true` requirement and the Codex fingerprint headers (User-Agent, originator, ChatGPT-Account-ID from JWT).
- Without seeing the actual response body of the live 400, we can't fully rule out a payload field changing. **This is the strongest argument for Phase 3 (surface the body) before Phase 2 (apply a fix).** Phase 3 will tell us whether the upstream message is "model not supported" (#1/#2) or something else.

## Recommended phase ordering (deviates from dispatch)

The dispatch suggests Phase 2 (fix) → Phase 3 (surface body) → Phase 4 (doctor). Flipping the first two saves a guess:

1. **Phase 3 first (surface the body).** ~20 LOC change to `ProviderError` constructor or to the throw sites in `codexResponsesAdapter.ts:365-381` — append a parsed-message tail. Re-run `hi` once. Whatever OpenAI's 400 body says **is** the root cause, no inference needed.
2. **Phase 2 second (apply the right fix).** With the body in hand:
   - If body says *"model not supported when using Codex with a ChatGPT account"* → root cause is #1 or #2 above. Fix: filter picker by entitlement (or at minimum, ship a `chatgpt-plus`-specific picker that hides `gpt-5` and any non-`-codex` standard variant unless the account is known to entitle them). Plus a one-shot config migration that resets `model.modelId` to `gpt-5.1-codex-max` when current value is `gpt-5`.
   - If body says something else → diagnose from there.
3. **Phase 4 last (doctor `--providers`).** Independent of root cause; provides ongoing monitoring.
